# 架构演进记录

本文不重复描述当前代码，而是保留 `mini-coding-agent` 从 A 到 B 的关键变化：旧设计是什么、暴露了什么问题、如何迁移、保留了哪些兼容层，以及下一步向哪里演进。

文档分工如下：

- [ARCHITECTURE.md](ARCHITECTURE.md) 只描述当前可验证的架构事实。
- 本文记录重要设计迁移，不保存每个类和每次重命名。
- [ROADMAP.md](ROADMAP.md) 记录尚未完成的方向，已落地内容应移到当前架构或本文。
- Git 历史保存完整代码细节；本文只保留理解设计决策所需的旧版本摘要。

以后发生较大架构调整时，至少补充“原设计、问题、新设计、迁移兼容、后续方向”五项。旧实现可以从主架构文档删除，但不能连设计原因一起消失。

## 演进总览

| 主题 | A：原设计 | B：当前设计 | 主要收益 |
|---|---|---|---|
| 任务执行 | Direct、Web、Review、Repository Analysis、AgentLoop 分开执行 | 单一 `AgentLoop` + `AgentTaskContract` | 生命周期、权限和完成性检查统一 |
| 审查与分析 | 两套高度相似的执行模块 | 共用 `REPOSITORY_INVESTIGATION`，只区分输出契约 | 消除重复调查逻辑 |
| 能力回答 | 模型根据当轮工具猜产品能力，局部问句特判 | `CapabilityRegistry` + 组合式意图识别 + `CapabilityTruthGuard` | 不再把“本轮未授权”说成“产品不支持” |
| 运行展示 | 零散日志、结束后输出大段 diff | `AgentRuntimeEvent` 时间线 + 可激活 Changes 卡片 + 终端 Diff Viewer | 执行可观察，代码改动按需查看 |
| 产物追问 | “在哪里”再次交给模型解释 | 从紧邻上一轮的 `FILE_CHANGE` 确定性解析 | 避免指代漂移 |
| 文件读取 | 单次固定行数读取 | Token 预算分页 + 行/列续读 + 文件哈希 + EOF 门禁 | 能完整审查大文件和超长单行，又不一次塞满窗口 |
| 会话压缩 | 字符阈值、关键行比例和最近尾部 | `structured-salience-v2` 分层选择、字符/Token 双预算和来源追踪 | 压缩结果更稳定、可解释、可审计 |
| 对话一致性 | 指示词命中后只保留最近一轮，质疑旧回答时可能删掉原话 | `PRIOR_RESPONSE_AUDIT` 检索 + 可观察选择策略 + 有界一致性重试 | 能依据会话原文承认和撤回旧说法，避免改写历史 |
| 外部事实 | Direct 依赖模型记忆，只有显式实时/Web 问法要求证据 | 前置 `ExternalFactPolicy` + Direct 草稿 `EvidenceRiskAssessor` + 动态契约升级 | 路由漏判也不能直接发布高风险事实；无工具或证据不足时不再强行补全 |
| 任务 Diff | 读取整个脏工作树 | 任务前后树快照 + 独立 Diff Artifact | 只展示当前任务真正产生的改动 |
| 运行存储 | 宽松权限、索引无锁、审计错误向业务冒泡 | owner-only 权限 + 文件锁/原子替换 + 尾部恢复 + best-effort 观察者 | 避免泄漏、并发丢计数和副作用重复 |
| 命令验证 | 根据命令字符串包含关系猜测试 | 结构化 executable/args 分类 | `echo npm test` 不能伪造验证通过 |

## 1. 多执行器到单一 AgentLoop

### A：按功能拆执行器

早期版本把普通回答、联网回答、代码审查、仓库分析和仓库修改分成独立 Task/Support 模块。这样容易快速实现单个入口，但随后出现了三类问题：

- 代码审查与仓库分析的读取、循环和错误处理几乎一致。
- Session、事件、上下文和 Guardrail 在不同路径上逐渐不一致。
- 新能力必须修改多个执行器，产品能力声明也容易与真实工具权限漂移。

### B：任务差异变成契约

现在所有 CLI 请求进入同一个 `AgentLoop`。`TaskContractBuilder` 把语义路由结果编译为 `AgentTaskContract`，契约声明：

- 本轮允许看到和调用的能力。
- 证据与成功结束条件。
- 单步或多步执行策略。
- 输出类型和步数预算。

Direct 仍使用单步补全，避免普通问候承担多轮结构化决策成本；但它的 Session、事件和完成生命周期没有另起一套。Review 与 Analysis 共享只读调查契约，只在 finding 或 architecture summary 的输出要求上不同。

迁移期间保留 `DIRECT_ANSWER`、`WEB_ANSWER`、`CODE_REVIEW` 等历史结果标签，新的记录同时带 `executionEngine`、`taskKind` 和 `outputKind`。兼容标签只服务旧 Session 和 Change Log，不再决定执行路径。

### 下一方向

继续让新能力表现为可组合的 Capability、Context Provider 和 Final Validator。只有状态机和副作用语义真正不同，才新增 Runtime。

## 2. 从“模型自述能力”到产品事实链

### A：当轮权限被误解为全局能力

Direct 契约不开放写文件或 Web 工具时，模型可能回答“我不能写文件/联网”。后续为了修复真实对话，又容易加入针对某几句问法的提前返回，形成难维护的句子白名单。

### B：Registry 是唯一事实源

当前由 `CapabilityRegistry` 描述产品支持的能力及边界。组合式分类器识别“产品主体 + 能力主题 + 疑问/解释意图”，而不是枚举完整句子。模型最终回答仍会经过 `CapabilityTruthGuard`；如果它把暂时的任务权限错误表述为全局能力缺失，运行时依据 Registry 纠正并记录事件。

这条迁移确立了一个边界：LLM 可以组织解释，但不能定义产品事实；Task Contract 表示本轮最小权限，也不是产品总能力清单。

### 下一方向

用多语言和开放表达 Eval 扩大组合式分类器覆盖，减少规则对固定措辞的依赖，同时继续以 Registry 作为最终事实锚点。

## 3. 从输出日志到可审计终端时间线

### A：过程与结果混在一起

早期输出难以区分 Conversation、Context、Prompt Cache 和本地缓存；命令执行过程不连续，完整 diff 直接铺在主时间线里也会淹没任务状态。

### B：同一事件协议，多种展示层级

AgentLoop 现在发布版本化 `AgentRuntimeEvent`。普通模式显示阶段和结果，`--verbose` 展开 Token、缓存和压缩统计，`--trace` 展开经过脱敏的结构化决策与选择原因。隐藏思维链不对外输出，终端展示的是显式计划、工具证据、Guardrail 和可复核的本地决策。

代码或文档变更只在时间线显示 Changes 卡片。用户激活卡片后进入终端 Diff Viewer；非交互环境可用 session id 再打开 diff。该实现保持纯 CLI 路线，没有引入 Web 控制台。

### 下一方向

基于同一事件协议增加持续式全屏 TUI、折叠长工具结果和 Session Replay，而不是为每种界面复制运行逻辑。

## 4. 从固定行数读取到完整文件覆盖

### A：一次最多读取固定行数

固定 800 行能保护上下文窗口，但把工具返回上限误当成了 Agent 的阅读上限。模型可能只看到文件开头就声称完成“完整审查”。

### B：分页是传输策略，覆盖率是完成条件

`read_file` 现在同时受行数和 Token 预算约束，并返回 `hasMore`、`nextStartLine`、`nextStartColumn`、总行数和内容哈希。普通内容按行继续；某一行本身超过预算时按列继续。`AgentState` 合并同一版本文件的完整行区间和部分行游标；代码审查和明确要求完整读取的任务，在覆盖第 1 行到 EOF 前不能成功结束。

源码不会被重复复制进 Checkpoint。最新读取块以高优先级 Context Section 注入，旧工具记录只留紧凑证据。文件变化后，旧哈希对应的覆盖状态自动失效。

### 下一方向

根据语法边界调整分页位置，并评估并行只读扫描与符号级索引；仍然坚持“能完整读完”不等于“一次把全文塞入模型窗口”。

## 5. 从全局脏 Diff 到任务级变更工件

### A：结束时读取整个 Working Tree

旧实现直接在任务结束时执行工作树 diff。用户在任务开始前已经存在的未提交修改会混进 Agent 的结果、Change Log 和后续记忆，数组顺序映射还可能把 diff 片段绑定到错误文件。

### B：任务前后快照

写任务启动时使用独立临时 Git index 捕获 before tree，结束时捕获 after tree，再由两个 tree 生成带文件路径映射的 `TaskDiffArtifact`。新增、修改、删除和重命名都按路径关联，不再按数组下标猜测。主时间线只显示 Changes 入口，用户主动打开后才渲染完整 diff。

### 下一方向

增加二进制文件预览策略和更细粒度的 hunk 选择；继续保持“展示工件”与“执行时间线”解耦。

## 6. 从宽松审计到可恢复的本地运行存储

### A：审计与业务成功耦合

旧实现中，工具或补丁已经产生副作用后，如果 Session/Event 写入失败，整个动作可能被报告为失败并被模型重试；`.mini-agent` 权限继承 umask，多个 Store 并发更新 `index.json` 也可能丢失计数。JSONL 尾部半条写入会让此前所有完整记录无法读取。

### B：副作用与观察者分层

当前目录和文件分别固定为 `0700`/`0600`，共享索引使用文件锁和原子替换。末尾不完整 JSONL 被视为可恢复尾巴，中间损坏仍报出精确行号。工具审计经过脱敏和体积限制，源码正文不在 Session/Event 中双份保存；观察者失败写入 warning 元数据，不反转已经成功的业务结果。

Checkpoint 恢复也从“同一 session 就恢复”收紧为“同一 session、模式和目标才恢复”，避免新任务继承旧副作用。

### 下一方向

评估 SQLite WAL 以替代 JSONL + JSON index，并增加崩溃注入和多进程压力测试。

## 7. 从字符串验证到结构化命令证据

### A：扫描显示字符串

旧验证逻辑可能因为命令文本里出现 `npm test` 就把 `echo npm test` 当成测试，Shell 字符串也让真实可执行文件与参数难以区分。

### B：验证分类绑定执行结构

`CommandRunner` 保留 `executable + args` 结构，`CommandClassification` 只认可白名单中的真实测试、构建、类型检查和 `git diff --check` 形式。Shell 模式和无关 executable 不再产生验证证据。高风险 executable、包发布/安装、危险 Git 子命令和 inline code 要求显式批准；子进程默认剥离宿主敏感环境变量。

### 下一方向

把项目级允许命令和验证范围迁移到显式 policy 配置，并为 monorepo 增加 package/path 级验证映射。

## 8. 上下文压缩 v1 到 structured-salience-v2

### A：字符驱动的关键行与最近尾部

旧压缩策略主要依赖：

- `maxChars` 字符阈值。
- 固定比例保留 summary、error、memory 和显式约束。
- 剩余空间保留最近尾部。
- `/compact` 与自动 Session Memory 压缩各自实现选择逻辑。

它简单、确定性强，但会暴露以下问题：

- 中英文的字符数与 Token 成本不一致。
- 超长 Tool JSON 可能挤掉真正重要的用户约束。
- 无法知道一条内容因为什么被保留或丢弃。
- 自动压缩与显式 `/compact` 的结果可能漂移。

### B：分层、双预算、可追溯

`StructuredCompactor` 现在把候选记录分成：

1. `PINNED`：用户硬约束、错误、任务结果和既有压缩记忆。
2. `CONVERSATION`：最近用户请求与助手回答。
3. `EVIDENCE`：工具、命令和验证证据。

每条记录同时具有优先级、最近性加权、保留原因和来源 id。压缩过程先为各层预留预算，再按全局优先级使用剩余空间；单条超大记录先做 head-tail 裁剪，重复内容先去重。最终结果同时满足字符和估算 Token 预算，并在正文保留 `source:<id>`。

自动 Session Memory 和 `/compact` 共用这一核心。`--verbose` 显示策略、固定项、裁剪数和丢弃数；`--trace` 进一步显示每个来源的分层、裁剪状态和选择原因。Prompt Cache、Embedding Cache、Conversation History 与 Context Compaction 仍是不同指标，不互相冒充。

当前压缩保持确定性，没有自动调用 LLM 总结。原因是模型总结会增加成本、延迟和不可复现性，也可能把未出现的事实写入记忆。现阶段先保证证据可回溯，再考虑可选的语义总结层。

### 下一方向

- 用模型供应商 tokenizer 替换启发式 Token 估算。
- 建立 Turn、Task、Session 多级摘要，减少重复压缩。
- 为可选 LLM 摘要增加原始来源引用、事实一致性检查和失败回退。
- 建立压缩 Eval，衡量约束召回率、任务事实保留率、压缩率和后续回答正确率。

## 9. 从破坏性最近轮聚焦到可审计的对话一致性

### A：指示词会删除旧回答证据

旧版为了解决“这个怎么样”“它在哪里”一类指代歧义，只要短请求命中“这个、那些、刚才”等词，就把已经受预算限制的 Conversation History 再切成紧邻的一个 user/assistant exchange。这个规则把“解释优先级”实现成了“删除其它历史”。

当用户说“这个说法哪来的，以及你之前提到的其它说法”时，请求虽然包含指示词，本质却是在审计助手此前的输出。如果真正被质疑的原话位于更早几轮，latest-only 切片会恰好删除它。模型随后只能看到一次局部纠正，可能声称“我之前并没有那么说”或重新解释原意；下一轮恢复较多历史后又突然承认，形成明显的上下文断裂和“狡辩”观感。

### B：选择策略、原话检索与有界重试

当前 Conversation 选择显式区分三种策略：

1. `RECENT_HISTORY`：普通请求使用受预算限制的最近历史。
2. `LATEST_REFERENT`：普通隐式指代只选择紧邻 exchange，避免旧主题参与当前指代；该策略仅在排除审计语义后使用。
3. `PRIOR_RESPONSE_AUDIT`：组合识别“助手归因 + 质疑/核查”言语行为，从完整的结构化会话记录中检索相关 assistant 原话。

Audit 检索使用当前争议中的显著词和中英文词元匹配此前 assistant 消息，优先保留命中原话、相邻用户问题、后续纠正和最近尾部，再按消息数与字符预算组装最终 prompt。没有可靠命中时回退到最近历史，而不是退化为单一 exchange。`ConversationTrace` 记录 `selectionStrategy` 和 `matchedAssistantMessages`；终端会显示 `prior-response audit` 以及命中的旧回答数量。

选择正确的上下文仍不能保证模型一定诚实处理原话。因此 `PriorResponseTruthGuard` 只检查一个运行时可确定的命题：草稿是否否认了当前可见记录中的相关 assistant 输出，或是否在已知历史不完整时武断地说“我没说过”。高置信冲突触发最多一次修订调用，修订上下文携带可见原文并要求区分：

- 会话记录能证明“这段文字此前确实由助手输出”。
- 会话记录不能证明“这段文字描述的外部事实为真”。

如果唯一一次修订仍然否认原话，或修订调用失败，运行时返回确定性的安全回退：承认记录中出现过的文字、撤回未核验部分，并拒绝继续编造替代细节。重试和最终替换分别写入 Guardrail/Event，避免静默改写结果，也避免无限自我反思循环。

### 边界与兼容

- 普通“你说的方案怎么做”只有助手归因、没有质疑行为，不进入 Audit。
- 完整可见记录里确实没有匹配原话时，Guard 不会强迫模型作虚假承认。
- 原始 Session 记录已经驻留内存，检索阶段遍历完整结构化会话；只有最终 prompt 有硬预算。“没有被选中”仍不能被解释为“从未说过”。
- 旧 assistant 输出属于对话审计证据，不是外部事实来源，也不能满足 Web、知识库或仓库证据门槛。

### 下一方向

建立跨语言、同义改写和长会话 Eval，衡量旧 claim 召回率、误触发率、重试成功率与额外 Token 成本；只有词面检索无法稳定覆盖时，再引入带来源定位的语义召回。

## 10. 从无一般事实证据策略到 ExternalFactPolicy

### A：详细事实仍由 Direct 自由补全

旧路由主要把“最新、实时、联网搜索”等显式请求交给 Web。一个问题即使要求精确地点、完整名单、具体年份、掉落物或版本 API，只要内容看起来不是实时信息，也可能进入 `DIRECT_RESPONSE`。模型在知识不完整时容易生成结构完整、语气确定但没有证据的清单；用户纠正其中一点后，它还可能只修补局部错误并继续发明相关细节。

这不是单纯的 Prompt 问题。系统没有先判断“这类回答需要什么级别的证据”，也就无法在工具缺失、搜索失败或来源不足时建立统一的拒答边界。

### B：先决定证据等级，再决定任务契约

`ExternalFactPolicy` 现在独立于具体主题，把请求分为：

- `NOT_EXTERNAL_FACT`：仓库任务、产品能力问题、可推导计算、创作请求和只问“此前说过什么”的会话审计。
- `GENERAL_KNOWLEDGE`：定义、原理、概览、非穷举比较和“有哪些知名/代表性例子”这类开放式代表样本，可以使用模型的一般知识，但仍需表达合理的不确定性。
- `VERIFICATION_REQUIRED`：精确属性、完整/逐项清单、易变化事实、明确要求来源或事实核查的主张，必须进入带外部证据的研究契约。

分类依据是“精确、穷举、时效、核验”等组合信号，不为某个游戏、作品、人物或某一句回归用例写主题白名单。它还把两种容易混淆的问题拆开：

```text
“你之前是否说过 X？” -> Conversation Record
“你之前说的 X 是否为真？” -> External Evidence
```

当策略要求验证时，TaskRouter 生成 `WEB_RESEARCH`，由现有 Web 证据门槛约束搜索、正文抓取和最终引用。如果工具没有配置、网络不可达、页面不可读或独立来源不足，正确结果是明确说明无法完成核验或证据不足；运行时不能把模型参数记忆、搜索摘要或旧会话说法伪装成本轮已验证事实。

`ExternalFactPolicy` 降低的是“本应核验却直接编造”的风险，不承诺消灭全部模型错误。`GENERAL_KNOWLEDGE` 仍可能不完整，Conversation Guard 也只维护说法前后一致，二者都不能替代领域来源。

### 当时保留的下一方向

这一阶段仍把正确性主要寄托在请求分类上，因此记录了“使用主题无关 factuality Eval 校准漏判和误判”的后续方向。第 15 节继续完成了这项迁移：即使前置分类漏掉隐晦精确事实，运行时也会审计实际草稿并动态升级契约。

## 11. 从“尽力搜到”为止到忠实范围与失败闭环

### A：查询改写、猜 URL 和守卫死锁

旧 Web 流程允许模型自由生成检索词。例如用户问“有哪些知名歌曲”，模型可能把查询改成“最知名歌曲 / most famous songs”。这在召回层面看似接近，却把“代表性例子”收窄成了排名问题。如果底层搜索传输失败，模型还可能翻译同一查询重试，随后猜测一个看似合理的 Wikipedia 或文档 URL。最终证据守卫要求“必须成功搜索”，即使模型已经准备明确报告网络失败，也会拒绝 `FINAL`，连续失败计数最终只留下笼统错误。

`fetch_url` 还暴露了一个独立的 Node 运行时兼容问题：自定义固定 DNS lookup 始终按单地址回调，而较新的 Node HTTP 客户端可能请求 `all: true` 的地址数组，最终产生 `Invalid IP address: undefined`。这不是页面本身的事实错误，而是 pinned transport 的回调形状错误。

### B：范围守恒、来源血缘和可完成的降级

当前 Web 链路增加三道约束：

1. `ExternalFactPolicy` 不再把“有哪些知名/著名/代表性作品”自动视为完整清单。没有实时或核验要求时，它可以留在 `GENERAL_KNOWLEDGE`；“全部、完整、逐项、第一章有哪些对象”等有界清单仍要求验证。
2. 显式 Web 研究的首个查询必须保留用户实体、范围和限定词。`WebResearchPolicy` 会拒绝在用户没有要求排名时加入 `最 / most / top / best / greatest`。后续查询可以增加同义词提高召回，但不能改变最终回答范围。
3. `fetch_url` 只接受用户给出的 URL，或成功 `web_search` 返回的精确 URL。搜索失败后猜 Wikipedia、文档或新闻地址会在执行前被 Guardrail 拒绝。

`web_search` 的 `auto` provider fallback 现在共享一个总超时预算，不再让每个 provider 各自消耗完整超时。底层错误会展开 `cause` 中的网络错误码和信息；一次 provider/transport 失败后，等价换词搜索会被拒绝。只要确实尝试过搜索并在最终答复中明确说明“来源不足 / 无法核验”，Web 契约即可正常完成，而不是与“必须成功搜索”的后置条件形成死锁。

`fetch_url` 的 pinned lookup 同时支持单地址和 `all: true` 数组回调，修复新 Node 版本下的 `undefined` 地址。它仍保持 DNS 重绑定和内网地址防护；修复回调形状不降低 SSRF 安全边界。

### 边界

- “实现了 Web 工具”不等于“当前机器网络一定可达”。无 API 的 DuckDuckGo HTML/Lite provider 仍依赖出站 DNS、TLS 和站点可用性，生产环境更适合接入稳定搜索 API。
- 搜索失败时只能回答“本轮无法核验”，不能偷偷退回模型记忆生成一个伪装成联网结果的清单。
- 普通 Direct 代表性列表仍属于未核验的一般知识；如果用户要求来源、完整性或排名，应明确进入 Web 契约。

### 下一方向

接入可配置的正式搜索 provider、记录 provider 健康状态和熔断窗口，并为 query fidelity、URL lineage、transport failure 和 graceful degradation 建立独立 Eval。

## 12. 从搜索引擎前五名到可验证的时效最高级

### A：DuckDuckGo 成功，但旧结果仍排在前面

真实会话中，用户询问某厂商“最新的模型”。`web_search` 网络请求成功，DuckDuckGo HTML provider 也解析出了 10 条候选，但工具按 provider 原始顺序先截成 5 条；旧版官方发布页排在第一，真正更新的发布页没有进入模型上下文。Agent 随后抓取旧官方页和一个陈旧第三方汇总，机械满足“两个页面、两个域名”后，把“曾经发布”错误推导成“截至今天仍是最新”。

这说明搜索质量是直接诱因，但不能把正确性责任交给搜索引擎排名。换成其它 provider 仍可能遇到索引延迟、SEO 排序或摘要陈旧；Agent 必须把搜索结果当候选，而不是已经排序好的事实。

### B：候选池重排、权威搜索与更高版本冲突

当前 `web_search` 将“候选召回数量”和“返回给模型的数量”分开：先收集 provider 已解析的完整有界候选池，再对时效查询进行本地重排，最后应用 `maxResults`。重排信号包括：

- 查询实体与结果域名的亲和度，用于提升可能的一手来源；
- 标题/摘要中的可见发布日期；
- release、changelog、models、updates 等发布型路径；
- 同一命名系列中的可比较版本号；
- 查询词与标题、URL、摘要的相关性。

随后又把实现边界进一步拆成三层：

1. `DuckDuckGoSearchProvider` 只保留 HTML/Lite 的 URL、网络请求和 DOM-like 文本解析，这是当前 provider 的局部技术债。
2. `WebSearchPipeline` 只依赖 `WebSearchProviderAdapter` 接口，统一处理总超时、fallback、候选归一化、跨 provider URL 去重、候选池上限和最终结果数量。
3. `WebSearchRanking`、`WebResearchPolicy` 与 Final Guardrail 不读取 provider 名称，不为 DuckDuckGo、OpenAI 或 GPT 写分支。

截图中的 DuckDuckGo Search Assist 能回答 GPT‑5.6，而传统网页结果仍把 GPT‑5.5 排在第一，恰好说明同一搜索产品的不同结果面也可能不一致。当前 HTML/Lite adapter 只能消费传统结果页，不能假设网页 UI 中的 Search Assist 一定存在于该端点。正确方向不是专门抓取这张卡片，而是让任意未来的正式 Search API、MCP 搜索或其它 provider adapter 都进入同一候选和证据管线。

重排只改善召回，不能单独证明“最新”。`TaskContractBuilder` 对“最新模型/版本/发布”要求至少两次非等价搜索，其中一次必须使用 `官方/official`、release notes、changelog 或 `site:` 约束核对当前来源。此类请求不再为凑数量强制两个独立域名；一个权威当前目录或正式发布页可以强于两个陈旧汇总，但不能只看一次搜索结果就推断“不存在更新版本”。

Final Guardrail 还会比较结论和本轮搜索/抓取证据中的命名版本。如果答案声称 `X-5.5` 最新，但证据里出现 `X-5.6`，系统不会武断判定 5.6 已发布，而是拒绝直接完成，要求先调查更高候选或明确报告来源冲突。这是“需要继续核验”的确定性边界，不是把版本号大小替代事实核验。

### 边界与下一方向

- 当前 first-party 识别是轻量域名亲和度和查询约束，不是完整组织域名知识图谱。
- 页面没有机器可读发布日期时，只能使用标题/摘要中的可见日期；正式搜索 API 的结构化时间字段会更可靠。
- 非数字命名、预览版/稳定版、区域性发布和产品层级仍需要模型结合权威正文判断。
- 后续应接入可配置正式搜索 provider，并记录 provider 原始 rank、本地 rank、日期和 authority signals，建立离线搜索质量 Eval。

## 13. 从查询关键词守卫到权威证据闭环

### A：已经搜索官方站点，仍被正则拒绝

一次“最新模型”研究已经执行 `site:官方域名 产品 2026`，并抓取多个官方产品页，但 Final Guardrail 仍报告“没有权威时效搜索”。旧判定要求同一个查询同时命中 `site:/official` 和 `latest/current/release/model` 等文字；当前年份不算时效信号。守卫提示又只说“使用 `site:`”，没有暴露这一隐藏条件。模型因此认为要求已经满足，连续重复 `FINAL`，最后被通用连续失败熔断。

这说明严格不等于堆叠关键词。只检查 query 会产生两类错误：把已经抓取的权威正文判为无证据，以及把包含 `official latest` 但实际返回第三方页面的查询误判为充分。

### B：验证查询、候选和正文的闭合血缘

现在时效最高级使用四段证据链：

```text
权威 + 时效检索意图
        ↓
检索实际返回的约束域名候选
        ↓
fetch_url 精确抓取其中一个候选
        ↓
正文出现版本、日期、发布、更新或 current/latest 证据
```

当前年份或相邻年份可以表达检索的时效意图，但不会单独证明某个结果最新。`site:example.com` 返回的站外噪声会被排除；抓取另一条普通搜索中的第三方页面不能替代官方候选检查；抓到只有公司介绍、没有版本或时间信息的页面也不能完成“最新”结论。更高同系列版本冲突检查仍在证据闭环之后执行。

Guardrail 的恢复提示同时改为动作级反馈，明确下一步必须调用 `web_search` 还是对精确候选执行 `fetch_url`。可恢复的守卫拒绝与模型、工具执行失败分别计数；如果模型持续忽略同一个守卫，最终错误会保留具体 guardrail code 和恢复原因，而不是退化成笼统的 `failed too many consecutive steps`。

### 边界与下一方向

- 没有正式 provider authority metadata 时，`official` 查询仍只能表达检索意图；`site:` 约束能提供更强的域名边界。
- 页面正文缺少结构化发布日期时，系统只能识别可见版本、日期和发布语义，不能推导隐藏发布时间。
- 后续 Search API 应保留原始 rank、发布时间、source type 和 authority signals，让同一证据链使用结构化字段而不是新增 provider 特例。

## 14. 从固定 10 步到有终止策略的研究预算

### A：证据已经接近充分，最后一步仍猜 URL

一次最新模型查询连续完成普通搜索、正文抓取、官方站点搜索和官方发布页抓取。第 9 步已经具备可回答的权威版本与日期证据，但模型在第 10 步猜测一个没有出现在搜索结果中的新闻目录 URL。URL 血缘守卫正确拦截，随后固定 `maxSteps=10` 立即耗尽，系统没有机会把已有证据整理成答案。

直接把上限调大只能降低偶发失败，不能阻止模型把新增步骤继续用于无边界搜索。根因是模型只能在提交 FINAL 后看到后置条件，不知道当前证据阶段、剩余预算和系统推荐的终止动作。

### B：研究进度、前置里程碑和综合预留

当前 Web 契约默认使用 14 次决策，并由 `WebResearchProgress` 在每轮上下文中输出：

- 当前阶段：DISCOVER、AUTHORITY_SEARCH、INSPECT_SOURCE、COMPARE_EVIDENCE 或 SYNTHESIZE；
- 搜索视角、抓取来源和独立域名的完成数量；
- 权威搜索、精确候选抓取和正文时效证据是否满足；
- 可用引用、剩余决策、证据是否已闭环；
- 唯一推荐的下一动作。

最新类任务完成第一个普通搜索视角后，后续 `web_search` 必须先转向官方、release、changelog 或 `site:` 时效查询，不能堆积更多普通结果页等待 Final Guardrail 纠正。默认 14 步中的最后 2 步是综合预留：模型侧不再暴露 Web tools，运行时也拒绝新的工具、计划或提问。证据满足时只能综合引用并回答；证据仍不足时必须透明失败或输出限制性结论。

这不是让确定性状态机替模型选择事实答案。模型仍负责生成查询、选择候选、解释不同产品系列和组织最终语言；状态机只约束里程碑、证据血缘、预算和停止条件。

### 边界与下一方向

- 14 是默认上限，不是完成配额；证据在第 5 步闭环时应立即回答。
- 最终综合预留不能修复质量差的来源，只保证任务不会把最后一步花在无法使用的新工具结果上。
- 正式 Search API 可以减少召回噪声并提供结构化日期、authority 和原始 rank，但仍需遵守同一进度与终止策略。

## 15. 从一次性事实路由到声明级动态证据升级

### A：请求分类一旦漏判，Direct 就没有恢复机会

`ExternalFactPolicy` 能覆盖大量“最新、谁是、多少、完整名单”等问题，但自然语言结构远多于有限规则。例如“第三章 Boss 是谁”使用后置“是谁”和序数范围；如果它没有命中前置策略，就会获得无 Web 工具、单步完成的 Direct 契约。模型即使在 2026 年运行，也可能输出“尚未发售、计划 2024 年发布”这样的过期说法。因为 Web 从未执行，这种失败与 DuckDuckGo 或 Search API 的质量无关。

继续给 Policy 增加游戏名、Boss、章节或某几句回归问法，只会形成领域白名单；把所有一般知识都强制联网，又会让问候、定义和原理说明承担不必要的延迟与失败率。前置分类适合作为风险先验，不适合作为唯一且不可逆的能力边界。

### B：先低成本起步，再审计实际声明并原地升级

新增 `EvidenceRiskAssessor`，在 Direct 候选 `FINAL` 对用户可见之前审计“模型实际准备发布什么”，而不只分析“用户最初怎么问”。它组合有界关系、精确属性、强否定、时态状态、运行日期矛盾和最近事实纠错等主题无关信号。普通定义和解释保持 Direct；仓库、产品元问题和可推导结论继续使用各自的本地证据；对旧错误的承认与撤回不会被误当成再次发布该事实。

高风险草稿被扣留后：

1. 草稿只写入审计记录，不加入 AgentState 的 Recent Decisions，也不写入对话历史。
2. 当前 `AgentTaskContract` 从 `DIRECT_RESPONSE` 原地升级为 `WEB_RESEARCH`。
3. 步数预算按升级后的研究契约扩展，下一轮重新构建 Context，并只开放 Web 工具。
4. 后续仍由已有搜索血缘、来源门槛、时效证据、综合预留和 Final Guardrail 约束。
5. 没有真实网络或证据不足时输出限制性结论，而不是回退到刚被扣留的模型记忆。

因此 `ExternalFactPolicy` 仍可让明显高风险请求直接进入 Web，节省一次 Direct 调用，但它已不再是正确性的单点依赖。整个修复没有引入游戏、人物、作品或搜索 provider 特例。

### 边界与下一方向

- 当前审计使用确定性、可测试的声明结构信号，不尝试把另一个 LLM 判分当作事实真相。
- 运行日期矛盾只拦截可机械证明的过去计划年份，不会自行断言产品后来究竟是否发布。
- 后续 Eval 应覆盖人物身份、产品版本、医学状态、赛事结果、作品章节和基础定义等跨领域样本，分别量化漏升级率、误升级率、额外 LLM 调用与网络失败时的可解释性。
- 更高质量 Search API 会提升升级后的召回率和来源元数据，但不会取代这条动态证据边界。

## 16. 从“证据够了就结束”到独立回答质量契约

### A：事实更安全了，但回答退化成最短合法摘要

Direct 草稿升级 Web 后，不再直接发布未经核实的精确事实，但真实试用暴露了下一层问题：模型执行一次搜索和一次抓取，刚达到来源数量门槛就返回一段很短的 `FINAL summary`。这不是输出 Token 被截断；调用的 `finishReason` 是 `stop`，生成量远低于 4096 Token 上限，并且短答在 `/compact` 之前已经出现。

原因有三类：

1. Web 契约规定了搜索、抓取、引用和终止条件，却没有表达定义、数量、枚举或解释题分别怎样才算“回答完整”。
2. Final Guardrail 只验证来源数量和 URL 血缘；一个 HTTP 200 的 WAF JSON、验证码或登录壳也可能被计为成功正文。
3. 路由仍残留“世界杯、股票”等主题名词，导致“世界杯是什么”这样的稳定定义题绕过 `GENERAL_KNOWLEDGE`，无意义地进入 Web。

短追问还存在语义丢失：“360”虽然能从会话被模型理解为“360 有多少子公司”，但契约仍按原始字符串构建，所以数量型答案要求和双来源门槛没有继承。

### B：证据质量、回答形态和追问语义分别建模

这轮增加了四条通用边界：

1. `AnswerQualityPolicy` 独立于证据来源，识别定义、数量、枚举、有界关系、身份、解释和一般回答，同时保留用户要求的简短、均衡或详细深度。
2. Task Contract 明确“达到证据数量只是最低条件”；Final Guardrail 按语义检查数量题是否给出数字或解释统计口径、枚举题是否提供列表、定义题是否真正定义对象，以及答案是否只有链接没有内容。系统不使用最低字符数。
3. `fetch_url` 把非 2xx、WAF、CAPTCHA、安全验证、访问拒绝和登录壳标记为失败；最终引用必须至少命中一个真正抓取过的 URL，搜索结果候选不等于已检查来源。
4. 主题名词不再单独触发 Web。“世界杯最新比分”仍因时效和比分进入研究，“世界杯是什么”保持 Direct。短追问会先恢复“有多少/有哪些”等谓词，再用完整问题重建原类型契约。

精确数量、精确属性、完整枚举和 Direct 风险升级后的有界事实默认要求两个独立正文来源。权威时效最高级继续使用专门的闭环，不机械叠加域名数量。

### 边界与下一方向

- 回答质量 Guardrail 只检查可确定的语义形态，不判断文章风格优美，也不为所有答案强制固定段落数。
- WAF 检测采用通用挑战页信号；后续应把无正文 SPA、地区限制、订阅墙和软 404 纳入离线页面质量数据集。
- 来源数量不能替代来源权威性。正式 Search API 提供 source type、发布时间和 authority metadata 后，应把“两个普通来源”升级为按声明风险选择来源组合。
- Eval 需要同时记录事实正确率、问题直接回答率、范围保持率、可用正文率和不必要 Web 路由率，避免只优化某一个指标。

## 17. 从多处关键词分流到统一语义控制平面

### A：修一个问句，另一类表达又误路由

旧实现虽然已经统一到一个 AgentLoop，但 `TaskRouter`、`ExternalFactPolicy`、`AnswerQualityPolicy`、Context 和 Follow-up 仍会分别解析原始字符串。长期补丁留下了几类典型问题：

- “项目管理”“file system”中的仓库词汇触发仓库模式；
- 创建普通文本或 `.mjs` 文件没有命中有限扩展名与动作表；
- `django` 被短语言名 `go` 命中，`websocket` 被 `web` 命中；
- 短追问继承了旧执行模式，却没有继承数量、枚举或关系谓词；
- 程序化调用方遗漏契约时获得了读写仓库和执行命令的宽权限默认值；
- Eval、CLI 和真实运行使用不同的隐含默认值，测试夹具掩盖了生产边界。

继续把作品名、算法题名、赛事名或完整中文问句加入生产判断，只会提高测试样例命中率，不会提高系统的组合泛化能力。

### B：TaskUnderstanding、显式契约与结构型回归

现在 `TaskUnderstanding` 为每个请求生成一份结构化记录：

```text
operation + target + answerShape + answerDepth
+ externalFactPolicy
+ explicitWeb / explicitRepositoryTarget / explicitMutation
+ completeFileRead + confidence + signals
```

TaskRouter 只映射兼容标签；TaskContractBuilder、ContextBuilder、EvidenceRiskAssessor 和 AnswerQualityPolicy 复用这份记录。默认契约改为 deny-by-default，AgentLoop 在调用方未传契约时也会自行构建同一语义契约。代码审查与仓库分析继续共享 `REPOSITORY_INVESTIGATION`，只读委派产生的路径证据可以满足父任务的仓库证据要求；完整文件审查仍必须由可验证的分页覆盖门禁保证。

长期记忆查询也不再维护另一套赛事、金融或算法主题分类器；`MemoryQueryBuilder` 根据同一 TaskUnderstanding 选择 Code、Review、Web、Conversation 或 General 召回偏好，只保留写入、运行、错误等检索扩展词。

短追问只做高置信度的谓词补全，并使用补全后的问题重新路由和构建契约；无法确定的省略交给带 Conversation 的模型，而不是猜一个作品、赛事或属性。文件类型识别改为词边界匹配，文件创建与修改识别使用动作和路径结构，不再维护具体算法名称。`SkillContextService` 等无调用死代码被删除；`--iterative` 成为描述执行策略的正式名称，旧 `--agent-loop` 仅保留兼容别名。

回归测试可以保留曾经出错的具体句子，因为测试样例不参与运行时。生产实现只允许产品事实、语义结构、权限和证据不变量进入确定性分支。测试同时加入词汇碰撞、同义改写、否定样例和默认最小权限，避免“只对原句通过”。

### 边界与下一方向

- 当前 TaskUnderstanding 仍是确定性规则组合，不是完整自然语言解析器；低置信度歧义应由 Conversation-aware 模型澄清或补全。
- AnswerQuality 的结构校验只覆盖可机械验证的数量、枚举、定义和来源空壳；事实正确性仍由证据链负责。
- 后续 Eval 应按语义维度生成释义和对抗样例，分别统计误路由、权限过宽、无谓联网、追问谓词丢失和回答形态失败，而不是按某个主题建立白名单。

## 18. 文档维护约定

一次重要架构迁移完成时：

1. 先更新 `ARCHITECTURE.md`，确保它与当前代码一致。
2. 在本文增加简短 A→B 记录，保留问题、取舍和兼容策略。
3. 从 `ROADMAP.md` 移除已经完成的事项，写入新的下一阶段。
4. 更新测试基线和相关测试计划。
5. 不把已删除类的完整接口长期复制进文档；需要实现细节时查 Git 历史。

这样既不会让主架构文档背负所有旧版本，也不会让项目只剩“现在长什么样”，却解释不了“为什么变成这样”。

## 19. 从只读实验开关到语义多 Agent 协作

### A：有实现，却在产品层等同于“没有”

旧版已经存在 `ReadonlySubAgentCoordinator`，但它不是完整产品能力：

- 默认 `enabled=false`，通常要靠 `--agents` 或配置开启；
- 模型只能输出 `DELEGATE_READONLY`，子 Agent 只能做本地只读调查；
- Capability Registry 和产品元意图不认识 subagent，所以“我们有 subagent 吗”可能被模型错误回答为没有；
- `auto` 只表示允许委派，不表示理解用户明确要求；代码审查与仓库分析的配置入口也不一致；
- writer → reviewer 这种常见工程流程无法表达。

这会把 Agent 能力退化成脚本参数：用户必须先知道隐藏开关，而自然语言目标本身不能驱动协作。

### B：自然语言意图、隔离提案和父级合入

现在多 Agent 进入 Capability Registry，并由 `SubAgentIntent` 组合识别 topic、问句、执行动作、禁用语义、数量、写入和审查信号。能力问答由本地事实源确定；明确执行请求进入仓库 Task Contract，并通过 Final Guardrail 保证真正发生委派。

协议从单一只读批次升级为：

```text
DELEGATE
  ├─ READ_ONLY
  ├─ PROPOSE_CHANGES
  └─ REVIEW_CHANGES (dependsOn writer)

parent review
  └─ APPLY_DELEGATED_PATCH
       └─ PatchManager + permission + verification
```

写入子 Agent 不获得主工作区写权限。它先读取证据，再返回 unified diff；协调器使用现有 `PatchManager.validatePatch` 做路径和 `git apply --check` 校验，仅把补丁提案放入父 Context。审查子 Agent 在依赖成功后读取仓库和提案，给出审查结果。主 Agent 必须显式选择合入，并承担最终验证，因此既支持真正的实现分工，也保留单写者的一致性边界。

默认策略改为可用而非强制使用。普通任务由模型判断是否值得拆分；用户明确要求时成为硬后置条件；`--agents` 只作为并发覆盖和兼容入口，`--agents 1`、配置 `off` 或自然语言禁用仍可以关闭。旧 `DELEGATE_READONLY` 只保留会话恢复兼容。

### 边界与下一方向

- 当前 writer 提供的是隔离补丁提案，不执行子级命令；编译、测试和最终验证仍由父 Agent 在合入后完成。
- reviewer 的意见作为父 Agent 的决策证据，不自动替代父级安全与完成性门禁。
- 下一阶段可以为补丁提案建立临时 worktree，使 writer 在不污染主工作区的前提下运行受控测试；还需要在 AgentBench 中统计委派收益、冲突率、review 拦截率和额外 Token/延迟。

## 20. 从委派失败黑盒到可恢复、可终止的协作闭环

### A：子任务失败、父级代写和后置条件死锁

一次真实的“用 subagent 创建独立 HTML 游戏”运行暴露了三个相互放大的问题：

1. 第一个 writer 已经返回新文件补丁，但协调器要求任何实现都先读取仓库证据，因而拒绝了本可独立验证的新文件提案。
2. 第二个 writer 读取仓库后返回非法 Decision JSON。底层客户端恢复失败后，协调器只发布 `FAILED` 状态，没有把精确协议错误显示到终端，也没有做任务级有界恢复。
3. 两个委派批次耗尽后，父 Agent 仍可普通 `APPLY_PATCH` 代写文件；最终 Guardrail 又坚持必须有 delegated proposal。模型在 `PLAN` 和被拒绝的 `FINAL` 之间切换，使“连续同一错误”计数被重置，最终一直运行到父级最大 Step。

文件最后存在并不等于任务正确完成：用户指定的执行主体没有完成工作，运行时却修改了主工作区，同时还以错误退出。

### B：协议恢复、主体边界、失败闭包和审计型可观察性

现在协作状态机按通用不变量收口：

- 新增文件补丁可以直接由 writer 提案；只要补丁只包含 `ADDED`、通过路径安全检查和 `git apply --check` 即可。修改或删除已有文件仍要求成功的仓库读取证据。
- 子级非法 JSON、空内容和常见 Schema 漂移进入共享预算约束下的有界恢复；终端显示 `recovery`、原始错误和下一动作。
- 子 Agent 每次调用模型前发布 `thinking step`，调用后发布结构化 Decision 类型与简短动作理由；工具、补丁提案和最终错误继续沿同一事件流输出。
- 用户明确指定 writer 时，父 Agent 在 delegated patch 合入前不能普通 patch 代写；明确要求 reviewer 时，没有完成的依赖评审也不能合入。
- 委派批次耗尽且硬性 writer/reviewer 条件仍未满足时，立即返回 `REQUIRED_DELEGATION_EXHAUSTED`，不再等待父级最大 Step，也不把代写伪装成协作成功。
- Partial 批次保留已完成 proposal，但不会清除失败原因；如果剩余预算不足以完成硬性评审，运行以未修改父工作区的失败结果结束。

“显示思考过程”被定义为可审计决策，而不是倾倒隐藏思维链。服务商可能返回 `reasoning_tokens` 和私有 `reasoning_content`，但后者并非稳定协议、可能包含内部草稿，也不应被当作可信执行日志。系统只记录其是否存在以及 Token 数；面向用户显示显式计划、Tool Call `reason`、子级 Decision、证据、Guardrail、恢复和错误。这样既能定位“在哪一步、因何失败、接下来做了什么”，又不会把不可验证的私有草稿混同于产品事实。

### 边界与下一方向

- 当前协议恢复针对可识别的输出格式错误；业务目标理解错误仍需要 Eval、任务重写或用户澄清，而不是无限重试。
- writer 仍只产生隔离补丁，不能在子级 worktree 中执行构建。后续临时 worktree 方案应继续保持父级合入唯一写边界。
- 可观察事件已经足以重建动作时间线；后续可以增加每个子任务的首字节延迟和分步 Token 增量，但不能因此持久化原始隐藏思维链。

## 21. 从补丁生成器到隔离开发者，从词面规则到混合语义层

### A：安全边界存在，但执行能力和语义覆盖不足

上一阶段 writer 只能读取父仓库并返回一段补丁。它不能观察补丁应用后的完整文件系统，不能运行测试，也不能根据失败继续修复；父工作区在子任务期间变化时，只会得到普通 `git apply` 失败，无法区分陈旧基线和无效补丁。

TaskUnderstanding 虽然消除了多路由器重复判断，但仍完全依赖确定性组合规则。明确的“修改文件”“不要联网”很稳定，条件任务、间接动作、复杂否定和跨轮指代仍可能被当成 Direct，或者因为否定句包含“修改”而得到过宽权限。

### B：一次性 worktree、验证闭环和模型辅助语义合并

现在每个子任务拥有独立工作区：

```text
parent current snapshot
  -> detached temporary worktree / isolated copy
  -> private baseline commit
  -> child APPLY_PATCH iterations
  -> allowlisted build/test commands
  -> diff against private baseline
  -> parent fingerprint + apply check
  -> explicit parent merge
```

Git worktree 会覆盖父级当时的 staged、unstaged 和非忽略 untracked 文件，因此子 Agent 看到的是用户真实工作状态，而不是只有 HEAD 的过时副本。writer 可以多次修改，并运行无 Shell、无安装、无联网的测试、类型检查、Lint 或 Build。失败命令回到子循环，最终提案记录验证证据。reviewer 在自己的工作区应用依赖 patch 后再检查。

提案携带父级基线指纹。父级合入时若指纹变化，会重新校验 patch；冲突返回 `DELEGATED_PATCH_CONFLICT`，要求重新委派，而不是覆盖并发修改。临时 worktree 在成功、失败和异常路径都会清理。

语义控制平面增加 `TaskUnderstandingResolver`。明确、简单请求和高置信度短追问仍走可复现的确定性路径；条件、复杂否定或间接动作会把可见 Conversation 一并交给模型并要求返回严格 JSON。模型负责补全语义，本地合并策略负责权限：

- 显式“不要修改”不能被模型升级为写任务；
- 明确 Web、产品状态、会话状态和已识别修改不能被模型降级；
- 非法、低置信度或矛盾 JSON 回退原确定性理解；
- 最终 Task Contract 和运行时 Guardrail 仍是实际权限边界。

这不是把路由完全交给模型，而是把开放式语言理解和确定性安全裁决分层。

### 边界与下一方向

- 临时 worktree 复用父级已有 `node_modules`，不允许子级安装依赖；缺失依赖时应返回可解释限制，由父级决定是否授权安装。
- 当前冲突策略是检测并重新委派，不自动生成三方冲突解决补丁。后续可在保持用户改动优先的前提下增加受审查的三方合并。
- 模型语义补全只在结构复杂时触发，仍需用跨语言释义、否定、条件和长会话 Eval 衡量额外 Token 成本与误授权率。
