# 架构演进记录

本文不重复描述当前代码，而是保留 `mini-coding-agent` 从 A 到 B 的关键变化：旧设计是什么、暴露了什么问题、如何迁移、保留了哪些兼容层，以及下一步向哪里演进。

文档分工如下：

- [ARCHITECTURE.md](ARCHITECTURE.md) 只描述当前可验证的架构事实。
- 本文记录重要设计迁移，不保存每个类和每次重命名。
- [ROADMAP.md](ROADMAP.md) 记录尚未完成的方向，已落地内容应移到当前架构或本文。
- Git 历史保存完整代码细节；本文只保留理解设计决策所需的旧版本摘要。

以后发生较大架构调整时，至少补充“原设计、问题、新设计、迁移兼容、后续方向”五项。旧实现可以从主架构文档删除，但不能连设计原因一起消失。

## 演进总览

| 主题 | A：原设计 | B：当前设计 | 主要收益 |
|---|---|---|---|
| 任务执行 | Direct、Web、Review、Repository Analysis、AgentLoop 分开执行 | 单一 `AgentLoop` + `AgentTaskContract` | 生命周期、权限和完成性检查统一 |
| 审查与分析 | 两套高度相似的执行模块 | 共用 `REPOSITORY_INVESTIGATION`，只区分输出契约 | 消除重复调查逻辑 |
| 能力回答 | 模型根据当轮工具猜产品能力，局部问句特判 | `CapabilityRegistry` + 组合式意图识别 + `CapabilityTruthGuard` | 不再把“本轮未授权”说成“产品不支持” |
| 运行展示 | 零散日志、结束后输出大段 diff | `AgentRuntimeEvent` 时间线 + 可激活 Changes 卡片 + 终端 Diff Viewer | 执行可观察，代码改动按需查看 |
| 产物追问 | “在哪里”再次交给模型解释 | 从紧邻上一轮的 `FILE_CHANGE` 确定性解析 | 避免指代漂移 |
| 文件读取 | 单次固定行数读取 | Token 预算分页 + 行/列续读 + 文件哈希 + EOF 门禁 | 能完整审查大文件和超长单行，又不一次塞满窗口 |
| 会话压缩 | 字符阈值、关键行比例和最近尾部 | `structured-salience-v2` 分层选择、字符/Token 双预算和来源追踪 | 压缩结果更稳定、可解释、可审计 |
| 对话一致性 | 指示词命中后只保留最近一轮，质疑旧回答时可能删掉原话 | `PRIOR_RESPONSE_AUDIT` 检索 + 可观察选择策略 + 有界一致性重试 | 能依据会话原文承认和撤回旧说法，避免改写历史 |
| 外部事实 | Direct 依赖模型记忆，只有显式实时/Web 问法要求证据 | `ExternalFactPolicy` 按精确度、完整性、时效性和核验意图选择证据等级 | 详细事实先取证；无工具或证据不足时不再强行补全 |
| 任务 Diff | 读取整个脏工作树 | 任务前后树快照 + 独立 Diff Artifact | 只展示当前任务真正产生的改动 |
| 运行存储 | 宽松权限、索引无锁、审计错误向业务冒泡 | owner-only 权限 + 文件锁/原子替换 + 尾部恢复 + best-effort 观察者 | 避免泄漏、并发丢计数和副作用重复 |
| 命令验证 | 根据命令字符串包含关系猜测试 | 结构化 executable/args 分类 | `echo npm test` 不能伪造验证通过 |

## 1. 多执行器到单一 AgentLoop

### A：按功能拆执行器

早期版本把普通回答、联网回答、代码审查、仓库分析和仓库修改分成独立 Task/Support 模块。这样容易快速实现单个入口，但随后出现了三类问题：

- 代码审查与仓库分析的读取、循环和错误处理几乎一致。
- Session、事件、上下文和 Guardrail 在不同路径上逐渐不一致。
- 新能力必须修改多个执行器，产品能力声明也容易与真实工具权限漂移。

### B：任务差异变成契约

现在所有 CLI 请求进入同一个 `AgentLoop`。`TaskContractBuilder` 把语义路由结果编译为 `AgentTaskContract`，契约声明：

- 本轮允许看到和调用的能力。
- 证据与成功结束条件。
- 单步或多步执行策略。
- 输出类型和步数预算。

Direct 仍使用单步补全，避免普通问候承担多轮结构化决策成本；但它的 Session、事件和完成生命周期没有另起一套。Review 与 Analysis 共享只读调查契约，只在 finding 或 architecture summary 的输出要求上不同。

迁移期间保留 `DIRECT_ANSWER`、`WEB_ANSWER`、`CODE_REVIEW` 等历史结果标签，新的记录同时带 `executionEngine`、`taskKind` 和 `outputKind`。兼容标签只服务旧 Session 和 Change Log，不再决定执行路径。

### 下一方向

继续让新能力表现为可组合的 Capability、Context Provider 和 Final Validator。只有状态机和副作用语义真正不同，才新增 Runtime。

## 2. 从“模型自述能力”到产品事实链

### A：当轮权限被误解为全局能力

Direct 契约不开放写文件或 Web 工具时，模型可能回答“我不能写文件/联网”。后续为了修复真实对话，又容易加入针对某几句问法的提前返回，形成难维护的句子白名单。

### B：Registry 是唯一事实源

当前由 `CapabilityRegistry` 描述产品支持的能力及边界。组合式分类器识别“产品主体 + 能力主题 + 疑问/解释意图”，而不是枚举完整句子。模型最终回答仍会经过 `CapabilityTruthGuard`；如果它把暂时的任务权限错误表述为全局能力缺失，运行时依据 Registry 纠正并记录事件。

这条迁移确立了一个边界：LLM 可以组织解释，但不能定义产品事实；Task Contract 表示本轮最小权限，也不是产品总能力清单。

### 下一方向

用多语言和开放表达 Eval 扩大组合式分类器覆盖，减少规则对固定措辞的依赖，同时继续以 Registry 作为最终事实锚点。

## 3. 从输出日志到可审计终端时间线

### A：过程与结果混在一起

早期输出难以区分 Conversation、Context、Prompt Cache 和本地缓存；命令执行过程不连续，完整 diff 直接铺在主时间线里也会淹没任务状态。

### B：同一事件协议，多种展示层级

AgentLoop 现在发布版本化 `AgentRuntimeEvent`。普通模式显示阶段和结果，`--verbose` 展开 Token、缓存和压缩统计，`--trace` 展开经过脱敏的结构化决策与选择原因。隐藏思维链不对外输出，终端展示的是显式计划、工具证据、Guardrail 和可复核的本地决策。

代码或文档变更只在时间线显示 Changes 卡片。用户激活卡片后进入终端 Diff Viewer；非交互环境可用 session id 再打开 diff。该实现保持纯 CLI 路线，没有引入 Web 控制台。

### 下一方向

基于同一事件协议增加持续式全屏 TUI、折叠长工具结果和 Session Replay，而不是为每种界面复制运行逻辑。

## 4. 从固定行数读取到完整文件覆盖

### A：一次最多读取固定行数

固定 800 行能保护上下文窗口，但把工具返回上限误当成了 Agent 的阅读上限。模型可能只看到文件开头就声称完成“完整审查”。

### B：分页是传输策略，覆盖率是完成条件

`read_file` 现在同时受行数和 Token 预算约束，并返回 `hasMore`、`nextStartLine`、`nextStartColumn`、总行数和内容哈希。普通内容按行继续；某一行本身超过预算时按列继续。`AgentState` 合并同一版本文件的完整行区间和部分行游标；代码审查和明确要求完整读取的任务，在覆盖第 1 行到 EOF 前不能成功结束。

源码不会被重复复制进 Checkpoint。最新读取块以高优先级 Context Section 注入，旧工具记录只留紧凑证据。文件变化后，旧哈希对应的覆盖状态自动失效。

### 下一方向

根据语法边界调整分页位置，并评估并行只读扫描与符号级索引；仍然坚持“能完整读完”不等于“一次把全文塞入模型窗口”。

## 5. 从全局脏 Diff 到任务级变更工件

### A：结束时读取整个 Working Tree

旧实现直接在任务结束时执行工作树 diff。用户在任务开始前已经存在的未提交修改会混进 Agent 的结果、Change Log 和后续记忆，数组顺序映射还可能把 diff 片段绑定到错误文件。

### B：任务前后快照

写任务启动时使用独立临时 Git index 捕获 before tree，结束时捕获 after tree，再由两个 tree 生成带文件路径映射的 `TaskDiffArtifact`。新增、修改、删除和重命名都按路径关联，不再按数组下标猜测。主时间线只显示 Changes 入口，用户主动打开后才渲染完整 diff。

### 下一方向

增加二进制文件预览策略和更细粒度的 hunk 选择；继续保持“展示工件”与“执行时间线”解耦。

## 6. 从宽松审计到可恢复的本地运行存储

### A：审计与业务成功耦合

旧实现中，工具或补丁已经产生副作用后，如果 Session/Event 写入失败，整个动作可能被报告为失败并被模型重试；`.mini-agent` 权限继承 umask，多个 Store 并发更新 `index.json` 也可能丢失计数。JSONL 尾部半条写入会让此前所有完整记录无法读取。

### B：副作用与观察者分层

当前目录和文件分别固定为 `0700`/`0600`，共享索引使用文件锁和原子替换。末尾不完整 JSONL 被视为可恢复尾巴，中间损坏仍报出精确行号。工具审计经过脱敏和体积限制，源码正文不在 Session/Event 中双份保存；观察者失败写入 warning 元数据，不反转已经成功的业务结果。

Checkpoint 恢复也从“同一 session 就恢复”收紧为“同一 session、模式和目标才恢复”，避免新任务继承旧副作用。

### 下一方向

评估 SQLite WAL 以替代 JSONL + JSON index，并增加崩溃注入和多进程压力测试。

## 7. 从字符串验证到结构化命令证据

### A：扫描显示字符串

旧验证逻辑可能因为命令文本里出现 `npm test` 就把 `echo npm test` 当成测试，Shell 字符串也让真实可执行文件与参数难以区分。

### B：验证分类绑定执行结构

`CommandRunner` 保留 `executable + args` 结构，`CommandClassification` 只认可白名单中的真实测试、构建、类型检查和 `git diff --check` 形式。Shell 模式和无关 executable 不再产生验证证据。高风险 executable、包发布/安装、危险 Git 子命令和 inline code 要求显式批准；子进程默认剥离宿主敏感环境变量。

### 下一方向

把项目级允许命令和验证范围迁移到显式 policy 配置，并为 monorepo 增加 package/path 级验证映射。

## 8. 上下文压缩 v1 到 structured-salience-v2

### A：字符驱动的关键行与最近尾部

旧压缩策略主要依赖：

- `maxChars` 字符阈值。
- 固定比例保留 summary、error、memory 和显式约束。
- 剩余空间保留最近尾部。
- `/compact` 与自动 Session Memory 压缩各自实现选择逻辑。

它简单、确定性强，但会暴露以下问题：

- 中英文的字符数与 Token 成本不一致。
- 超长 Tool JSON 可能挤掉真正重要的用户约束。
- 无法知道一条内容因为什么被保留或丢弃。
- 自动压缩与显式 `/compact` 的结果可能漂移。

### B：分层、双预算、可追溯

`StructuredCompactor` 现在把候选记录分成：

1. `PINNED`：用户硬约束、错误、任务结果和既有压缩记忆。
2. `CONVERSATION`：最近用户请求与助手回答。
3. `EVIDENCE`：工具、命令和验证证据。

每条记录同时具有优先级、最近性加权、保留原因和来源 id。压缩过程先为各层预留预算，再按全局优先级使用剩余空间；单条超大记录先做 head-tail 裁剪，重复内容先去重。最终结果同时满足字符和估算 Token 预算，并在正文保留 `source:<id>`。

自动 Session Memory 和 `/compact` 共用这一核心。`--verbose` 显示策略、固定项、裁剪数和丢弃数；`--trace` 进一步显示每个来源的分层、裁剪状态和选择原因。Prompt Cache、Embedding Cache、Conversation History 与 Context Compaction 仍是不同指标，不互相冒充。

当前压缩保持确定性，没有自动调用 LLM 总结。原因是模型总结会增加成本、延迟和不可复现性，也可能把未出现的事实写入记忆。现阶段先保证证据可回溯，再考虑可选的语义总结层。

### 下一方向

- 用模型供应商 tokenizer 替换启发式 Token 估算。
- 建立 Turn、Task、Session 多级摘要，减少重复压缩。
- 为可选 LLM 摘要增加原始来源引用、事实一致性检查和失败回退。
- 建立压缩 Eval，衡量约束召回率、任务事实保留率、压缩率和后续回答正确率。

## 9. 从破坏性最近轮聚焦到可审计的对话一致性

### A：指示词会删除旧回答证据

旧版为了解决“这个怎么样”“它在哪里”一类指代歧义，只要短请求命中“这个、那些、刚才”等词，就把已经受预算限制的 Conversation History 再切成紧邻的一个 user/assistant exchange。这个规则把“解释优先级”实现成了“删除其它历史”。

当用户说“这个说法哪来的，以及你之前提到的其它说法”时，请求虽然包含指示词，本质却是在审计助手此前的输出。如果真正被质疑的原话位于更早几轮，latest-only 切片会恰好删除它。模型随后只能看到一次局部纠正，可能声称“我之前并没有那么说”或重新解释原意；下一轮恢复较多历史后又突然承认，形成明显的上下文断裂和“狡辩”观感。

### B：选择策略、原话检索与有界重试

当前 Conversation 选择显式区分三种策略：

1. `RECENT_HISTORY`：普通请求使用受预算限制的最近历史。
2. `LATEST_REFERENT`：普通隐式指代只选择紧邻 exchange，避免旧主题参与当前指代；该策略仅在排除审计语义后使用。
3. `PRIOR_RESPONSE_AUDIT`：组合识别“助手归因 + 质疑/核查”言语行为，从完整的结构化会话记录中检索相关 assistant 原话。

Audit 检索使用当前争议中的显著词和中英文词元匹配此前 assistant 消息，优先保留命中原话、相邻用户问题、后续纠正和最近尾部，再按消息数与字符预算组装最终 prompt。没有可靠命中时回退到最近历史，而不是退化为单一 exchange。`ConversationTrace` 记录 `selectionStrategy` 和 `matchedAssistantMessages`；终端会显示 `prior-response audit` 以及命中的旧回答数量。

选择正确的上下文仍不能保证模型一定诚实处理原话。因此 `PriorResponseTruthGuard` 只检查一个运行时可确定的命题：草稿是否否认了当前可见记录中的相关 assistant 输出，或是否在已知历史不完整时武断地说“我没说过”。高置信冲突触发最多一次修订调用，修订上下文携带可见原文并要求区分：

- 会话记录能证明“这段文字此前确实由助手输出”。
- 会话记录不能证明“这段文字描述的外部事实为真”。

如果唯一一次修订仍然否认原话，或修订调用失败，运行时返回确定性的安全回退：承认记录中出现过的文字、撤回未核验部分，并拒绝继续编造替代细节。重试和最终替换分别写入 Guardrail/Event，避免静默改写结果，也避免无限自我反思循环。

### 边界与兼容

- 普通“你说的方案怎么做”只有助手归因、没有质疑行为，不进入 Audit。
- 完整可见记录里确实没有匹配原话时，Guard 不会强迫模型作虚假承认。
- 原始 Session 记录已经驻留内存，检索阶段遍历完整结构化会话；只有最终 prompt 有硬预算。“没有被选中”仍不能被解释为“从未说过”。
- 旧 assistant 输出属于对话审计证据，不是外部事实来源，也不能满足 Web、知识库或仓库证据门槛。

### 下一方向

建立跨语言、同义改写和长会话 Eval，衡量旧 claim 召回率、误触发率、重试成功率与额外 Token 成本；只有词面检索无法稳定覆盖时，再引入带来源定位的语义召回。

## 10. 从无一般事实证据策略到 ExternalFactPolicy

### A：详细事实仍由 Direct 自由补全

旧路由主要把“最新、实时、联网搜索”等显式请求交给 Web。一个问题即使要求精确地点、完整名单、具体年份、掉落物或版本 API，只要内容看起来不是实时信息，也可能进入 `DIRECT_RESPONSE`。模型在知识不完整时容易生成结构完整、语气确定但没有证据的清单；用户纠正其中一点后，它还可能只修补局部错误并继续发明相关细节。

这不是单纯的 Prompt 问题。系统没有先判断“这类回答需要什么级别的证据”，也就无法在工具缺失、搜索失败或来源不足时建立统一的拒答边界。

### B：先决定证据等级，再决定任务契约

`ExternalFactPolicy` 现在独立于具体主题，把请求分为：

- `NOT_EXTERNAL_FACT`：仓库任务、产品能力问题、可推导计算、创作请求和只问“此前说过什么”的会话审计。
- `GENERAL_KNOWLEDGE`：定义、原理、概览、非穷举比较和“有哪些知名/代表性例子”这类开放式代表样本，可以使用模型的一般知识，但仍需表达合理的不确定性。
- `VERIFICATION_REQUIRED`：精确属性、完整/逐项清单、易变化事实、明确要求来源或事实核查的主张，必须进入带外部证据的研究契约。

分类依据是“精确、穷举、时效、核验”等组合信号，不为某个游戏、作品、人物或某一句回归用例写主题白名单。它还把两种容易混淆的问题拆开：

```text
“你之前是否说过 X？” -> Conversation Record
“你之前说的 X 是否为真？” -> External Evidence
```

当策略要求验证时，TaskRouter 生成 `WEB_RESEARCH`，由现有 Web 证据门槛约束搜索、正文抓取和最终引用。如果工具没有配置、网络不可达、页面不可读或独立来源不足，正确结果是明确说明无法完成核验或证据不足；运行时不能把模型参数记忆、搜索摘要或旧会话说法伪装成本轮已验证事实。

`ExternalFactPolicy` 降低的是“本应核验却直接编造”的风险，不承诺消灭全部模型错误。`GENERAL_KNOWLEDGE` 仍可能不完整，Conversation Guard 也只维护说法前后一致，二者都不能替代领域来源。

### 下一方向

用主题无关的 factuality Eval 持续校准分类阈值，重点观察两类代价：把普通概念或代表性例子过度路由到 Web，以及让精确但措辞隐晦的事实请求漏过验证。未来若增加专用搜索 API，只替换证据提供者，不改变 Policy 和契约边界。

## 11. 从“尽力搜到”为止到忠实范围与失败闭环

### A：查询改写、猜 URL 和守卫死锁

旧 Web 流程允许模型自由生成检索词。例如用户问“有哪些知名歌曲”，模型可能把查询改成“最知名歌曲 / most famous songs”。这在召回层面看似接近，却把“代表性例子”收窄成了排名问题。如果底层搜索传输失败，模型还可能翻译同一查询重试，随后猜测一个看似合理的 Wikipedia 或文档 URL。最终证据守卫要求“必须成功搜索”，即使模型已经准备明确报告网络失败，也会拒绝 `FINAL`，连续失败计数最终只留下笼统错误。

`fetch_url` 还暴露了一个独立的 Node 运行时兼容问题：自定义固定 DNS lookup 始终按单地址回调，而较新的 Node HTTP 客户端可能请求 `all: true` 的地址数组，最终产生 `Invalid IP address: undefined`。这不是页面本身的事实错误，而是 pinned transport 的回调形状错误。

### B：范围守恒、来源血缘和可完成的降级

当前 Web 链路增加三道约束：

1. `ExternalFactPolicy` 不再把“有哪些知名/著名/代表性作品”自动视为完整清单。没有实时或核验要求时，它可以留在 `GENERAL_KNOWLEDGE`；“全部、完整、逐项、第一章有哪些对象”等有界清单仍要求验证。
2. 显式 Web 研究的首个查询必须保留用户实体、范围和限定词。`WebResearchPolicy` 会拒绝在用户没有要求排名时加入 `最 / most / top / best / greatest`。后续查询可以增加同义词提高召回，但不能改变最终回答范围。
3. `fetch_url` 只接受用户给出的 URL，或成功 `web_search` 返回的精确 URL。搜索失败后猜 Wikipedia、文档或新闻地址会在执行前被 Guardrail 拒绝。

`web_search` 的 `auto` provider fallback 现在共享一个总超时预算，不再让每个 provider 各自消耗完整超时。底层错误会展开 `cause` 中的网络错误码和信息；一次 provider/transport 失败后，等价换词搜索会被拒绝。只要确实尝试过搜索并在最终答复中明确说明“来源不足 / 无法核验”，Web 契约即可正常完成，而不是与“必须成功搜索”的后置条件形成死锁。

`fetch_url` 的 pinned lookup 同时支持单地址和 `all: true` 数组回调，修复新 Node 版本下的 `undefined` 地址。它仍保持 DNS 重绑定和内网地址防护；修复回调形状不降低 SSRF 安全边界。

### 边界

- “实现了 Web 工具”不等于“当前机器网络一定可达”。无 API 的 DuckDuckGo HTML/Lite provider 仍依赖出站 DNS、TLS 和站点可用性，生产环境更适合接入稳定搜索 API。
- 搜索失败时只能回答“本轮无法核验”，不能偷偷退回模型记忆生成一个伪装成联网结果的清单。
- 普通 Direct 代表性列表仍属于未核验的一般知识；如果用户要求来源、完整性或排名，应明确进入 Web 契约。

### 下一方向

接入可配置的正式搜索 provider、记录 provider 健康状态和熔断窗口，并为 query fidelity、URL lineage、transport failure 和 graceful degradation 建立独立 Eval。

## 12. 从搜索引擎前五名到可验证的时效最高级

### A：DuckDuckGo 成功，但旧结果仍排在前面

真实会话中，用户询问某厂商“最新的模型”。`web_search` 网络请求成功，DuckDuckGo HTML provider 也解析出了 10 条候选，但工具按 provider 原始顺序先截成 5 条；旧版官方发布页排在第一，真正更新的发布页没有进入模型上下文。Agent 随后抓取旧官方页和一个陈旧第三方汇总，机械满足“两个页面、两个域名”后，把“曾经发布”错误推导成“截至今天仍是最新”。

这说明搜索质量是直接诱因，但不能把正确性责任交给搜索引擎排名。换成其它 provider 仍可能遇到索引延迟、SEO 排序或摘要陈旧；Agent 必须把搜索结果当候选，而不是已经排序好的事实。

### B：候选池重排、权威搜索与更高版本冲突

当前 `web_search` 将“候选召回数量”和“返回给模型的数量”分开：先收集 provider 已解析的完整有界候选池，再对时效查询进行本地重排，最后应用 `maxResults`。重排信号包括：

- 查询实体与结果域名的亲和度，用于提升可能的一手来源；
- 标题/摘要中的可见发布日期；
- release、changelog、models、updates 等发布型路径；
- 同一命名系列中的可比较版本号；
- 查询词与标题、URL、摘要的相关性。

随后又把实现边界进一步拆成三层：

1. `DuckDuckGoSearchProvider` 只保留 HTML/Lite 的 URL、网络请求和 DOM-like 文本解析，这是当前 provider 的局部技术债。
2. `WebSearchPipeline` 只依赖 `WebSearchProviderAdapter` 接口，统一处理总超时、fallback、候选归一化、跨 provider URL 去重、候选池上限和最终结果数量。
3. `WebSearchRanking`、`WebResearchPolicy` 与 Final Guardrail 不读取 provider 名称，不为 DuckDuckGo、OpenAI 或 GPT 写分支。

截图中的 DuckDuckGo Search Assist 能回答 GPT‑5.6，而传统网页结果仍把 GPT‑5.5 排在第一，恰好说明同一搜索产品的不同结果面也可能不一致。当前 HTML/Lite adapter 只能消费传统结果页，不能假设网页 UI 中的 Search Assist 一定存在于该端点。正确方向不是专门抓取这张卡片，而是让任意未来的正式 Search API、MCP 搜索或其它 provider adapter 都进入同一候选和证据管线。

重排只改善召回，不能单独证明“最新”。`TaskContractBuilder` 对“最新模型/版本/发布”要求至少两次非等价搜索，其中一次必须使用 `官方/official`、release notes、changelog 或 `site:` 约束核对当前来源。此类请求不再为凑数量强制两个独立域名；一个权威当前目录或正式发布页可以强于两个陈旧汇总，但不能只看一次搜索结果就推断“不存在更新版本”。

Final Guardrail 还会比较结论和本轮搜索/抓取证据中的命名版本。如果答案声称 `X-5.5` 最新，但证据里出现 `X-5.6`，系统不会武断判定 5.6 已发布，而是拒绝直接完成，要求先调查更高候选或明确报告来源冲突。这是“需要继续核验”的确定性边界，不是把版本号大小替代事实核验。

### 边界与下一方向

- 当前 first-party 识别是轻量域名亲和度和查询约束，不是完整组织域名知识图谱。
- 页面没有机器可读发布日期时，只能使用标题/摘要中的可见日期；正式搜索 API 的结构化时间字段会更可靠。
- 非数字命名、预览版/稳定版、区域性发布和产品层级仍需要模型结合权威正文判断。
- 后续应接入可配置正式搜索 provider，并记录 provider 原始 rank、本地 rank、日期和 authority signals，建立离线搜索质量 Eval。

## 13. 文档维护约定

一次重要架构迁移完成时：

1. 先更新 `ARCHITECTURE.md`，确保它与当前代码一致。
2. 在本文增加简短 A→B 记录，保留问题、取舍和兼容策略。
3. 从 `ROADMAP.md` 移除已经完成的事项，写入新的下一阶段。
4. 更新测试基线和相关测试计划。
5. 不把已删除类的完整接口长期复制进文档；需要实现细节时查 Git 历史。

这样既不会让主架构文档背负所有旧版本，也不会让项目只剩“现在长什么样”，却解释不了“为什么变成这样”。
