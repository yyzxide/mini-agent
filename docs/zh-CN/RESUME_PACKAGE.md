# 简历包装与求职使用说明

这份文档只做一件事：

> 把 `mini-coding-agent` 从“我做了个项目”变成“我可以拿去投递和面试”的材料。

适用人群尤其是你这种情况：

- 目前主要经历偏客户端
- 想往更通用的软件工程、后端、平台、AI 应用工程方向跳
- 需要一个能讲出工程深度、又不至于吹过头的项目

## 1. 这个项目适合怎么定位

最稳的定位不是：

> 我做了一个对标 Claude Code / Codex 的完整工业级智能编程平台

而是：

> 我做了一个本地运行的 AI Coding Agent CLI，重点解决任务路由、工具调用、上下文记忆、补丁应用、命令执行、安全边界和执行审计问题。

这两句话看起来差不多，风险完全不同。

前者容易被追问到穿。  
后者真实、工程味足，而且已经足够体现能力。

## 2. 简历上一句话版本

### 版本 A：偏通用后端 / 平台 / 工程化

> 基于 TypeScript / Node.js 实现本地 AI Coding Agent CLI，支持自然语言任务路由、代码搜索、文件读取、补丁应用、命令执行、会话记录与运行审计，构建了受控工具系统、权限边界与 JSONL session/event 体系。

补充亮点：实现声明式 Skill 发现与上下文注入、覆盖所有执行模式的可控长期记忆，以及运行时强制只读的 `/plan` → `/execute` 规划执行闭环。

### 版本 B：偏 AI 应用工程

> 独立实现本地 AI Coding Agent，围绕大模型结构化决策、工具调用、上下文压缩、长期记忆检索、联网资料检索、代码审查和 Agent Harness 评测流程，完成从任务理解到执行审计的闭环工程落地。

### 版本 C：偏“从客户端跳出去”的转型表达

> 在客户端开发之外，独立完成一个 AI Coding Agent CLI 项目，重点提升后端工程化、Agent Workflow、模型集成、工具系统设计和执行安全控制能力。

## 3. 简历上两到三行版本

你可以直接抄这个，再按版面裁剪：

> `mini-coding-agent`：基于 TypeScript / Node.js 开发的本地 AI Coding Agent CLI。支持普通问答、联网问答、代码审查与仓库任务四类模式分流，具备 `ToolRegistry`、`AgentLoop`、`PatchManager`、`CommandRunner`、`SessionStore`、`LongTermMemoryStore` 等核心模块；实现 MCP stdio/Streamable HTTP 工具发现与调用、Web 证据闸门、可替换 embedding 的长期记忆治理、受控补丁与命令执行及 JSONL 审计，并通过 Vitest 覆盖核心链路。

## 4. 项目亮点怎么挑

如果简历空间有限，优先写这 5 个：

1. **多模式任务路由**  
   普通问答、联网问答、代码审查、仓库修改任务分流，对应 `DIRECT_ANSWER`、`WEB_ANSWER`、`CODE_REVIEW`、`AGENT_LOOP`。

2. **受控工具系统**  
   搜索代码、读文件、抓网页、打 patch、跑命令都走结构化工具和 zod 校验，并带只读、破坏性、幂等性、外部访问等能力标注。

3. **会话记忆与审计**  
   本地 JSONL session/event/log/change-log，加轻量长期记忆检索，可恢复、可追踪、可复盘。

4. **安全边界**  
   路径限制、`.git` / `.mini-agent` 内部路径拒绝、patch check、Git 换行配置隔离、命令超时、危险命令拦截、联网目标限制。

5. **评测与扩展性**  
   Agent Harness 支持脚本化多步场景、步骤/工具选择指标与失败归因；MCP 支持真实 tools runtime；当前正常环境回归基线为 39 个测试文件、288 个测试用例。

## 5. 面试自我介绍怎么接这个项目

### 30 秒版本

> 我最近重点做了一个本地 AI Coding Agent CLI 项目。它不是简单调大模型 API，而是把任务路由、工具调用、上下文记忆、补丁应用、命令执行和本地审计串成了完整闭环。这个项目对我最大的提升，是让我开始系统性地做后端工程组织和 AI Agent 工程化。

### 1 分钟版本

> 我做了一个本地运行的 AI Coding Agent CLI，名字叫 `mini-coding-agent`。用户可以在任意 git 仓库里直接下自然语言任务，系统先做任务分流：普通问答、联网资料问题、文件级代码审查和真正的仓库修改任务分别走不同路径。真正进入 AgentLoop 后，模型只负责输出结构化决策，本地 TypeScript 代码负责执行工具、应用 patch、运行命令和记录 session/event/log。整个项目让我把模型集成、工具系统、安全控制、执行审计和测试回归这些能力串起来了。

### 3 分钟版本

建议顺序：

1. 先说问题：普通聊天模型不能真正稳定执行仓库任务。
2. 再说方案：我做了本地 CLI Agent，不依赖后端和页面，先把核心闭环跑通。
3. 再说架构：CLI、TaskRouter、AgentLoop、ToolRegistry、CommandRunner、PatchManager、SessionStore。
4. 再说难点：结构化决策、路径安全、命令安全、上下文压缩、联网回答质量。
5. 再说结果：能完成本地代码任务闭环，当前正常环境回归基线为 39 个测试文件、288 个测试用例，可写入简历。

## 7. 面试官最可能关心的点

他们一般不会先问你“模型参数是多少”，而会问：

1. 为什么要做这个项目？
2. 这个项目和普通聊天机器人有什么区别？
3. 模型会不会乱改文件？
4. 你怎么做工具调用和安全控制？
5. 你怎么处理上下文记忆？
6. 为什么说它是 Agent，而不只是脚本？
7. 你自己最满意和最不满意的地方是什么？

这些问题，你现在已经有材料能答。

## 8. 哪些地方千万别吹

下面这些点不要包装过头：

- 不要说“完全对标 Claude Code / Codex”
- 不要说“已经达到生产级”
- 不要说“具备成熟生产级 RAG”
- 不要说“完整覆盖 MCP 全协议”；准确说法是“实现 MCP stdio/Streamable HTTP tools runtime”
- 不要说“具备高可靠实时联网搜索”
- 不要说“可以稳定自主完成任意复杂任务”
- 不要说“已经能替代真实 IDE Agent 或成熟商业 Coding Agent”

更稳的说法是：

- 这是作品级、工程化导向的本地 Agent MVP
- 核心闭环完整，能力边界清晰
- 重点展示架构设计、工具执行、安全边界、上下文管理和轻量本地长期记忆
- 当前主要价值在工程闭环和可回归性，不是模型算法创新

## 9. 如果面试官质疑“你原来是客户端，为什么做这个”

你可以这样答：

> 因为我不想把自己限制在单一端能力里。这个项目对我来说不是为了追热点，而是我刻意用一个 Agent 工程项目把后端工程化、系统设计、模型接入、任务编排和安全控制串起来。它既能体现我学习迁移能力，也能证明我不仅会写界面，还能把复杂执行系统搭起来。

这个回答很重要。
